UI SearchYCbCrRegularized Alternating Optimization

色重み探索は、
λと割り当てを同時に考える問題。

以前の説明では、複数の λC を試して再構築スコアで選ぶ二段階探索として書いていました。これはプロトタイプとしては使えますが、研究方針としては少し弱いです。最終的には、合成駒の組み合わせ π と色重み λC を、同じ目的関数の中でバランスさせる形に寄せます。

重要なのは、λCをただ自由に最小化しないことです。制約や正則化なしに λC を目的関数へ入れると、色コストを無視する λC=0 に逃げやすくなります。

1. 危ない定式化

単純に次のように書くと、目的関数は λC を小さくする方向に逃げます。

min_{π, λC}  P_Y(π) + λC P_C(π)

この式では、色の項 P_C が非負なので、λC を 0 にすれば目的関数が小さくなります。つまり「色を見ない」ことが最適化上は有利になります。

そのため、λC を最適化するなら、範囲制約、色寄与率の正則化、基準値からのズレの罰則が必要です。

2. 修正版の目的関数

まずは再配置変数を入れすぎず、合成駒の組み合わせ π と色重み λC を変数にします。

min_{π, λC} J(π, λC)

J(π, λC)
= α P_hat(π, λC)
+ (1 - α) B_hat(π)
+ β (s(π, λC) - τ)^2
+ γ (λC - λ0)^2

subject to
π_i is a permutation
λ_min <= λC <= λ_max

ここで、P_hat は合成駒内部のばらつき、B_hat は見た目側の補助評価、s は色寄与率です。

3. 合成駒内部コスト P_hat

画像 i の割り当てを π_i とし、合成駒 k に入るセル特徴量の平均との差を見ます。特徴量は YCbCr に分けます。

P_hat(π, λC)
= P_hat_Y(π)
+ λC { P_hat_Cb(π) + P_hat_Cr(π) }

Y

明るさ。形、輪郭、濃淡を担うため主成分として扱います。

Cb / Cr

色味。色に引っ張られすぎないよう、λCで補助的に効かせます。

4. 色寄与率の正則化

λC が 0 に逃げないよう、目的関数の中に色寄与率の目標を入れます。

s(π, λC)
= λC { P_hat_Cb(π) + P_hat_Cr(π) }
  / { P_hat_Y(π) + λC(P_hat_Cb(π)+P_hat_Cr(π)) + ε }

penalty = β (s(π, λC) - τ)^2

τ は目標となる色寄与率です。たとえば τ=0.25 とすれば、探索コストのうち色がだいたい25%程度効くように誘導します。

5. λの暴れ防止

画像セットによっては、色差が極端に小さかったり大きかったりします。その場合、λC が端に寄りすぎる可能性があります。

regularization = γ (λC - λ0)^2

example:
λ0 = 0.35
λ_min = 0.10
λ_max = 0.80

基準値 λ0 から離れすぎると罰則をかけることで、色の扱いを安定させます。

6. 見た目側の B_hat は別特徴を見る

再構築側の評価を、合成駒内部コストと同じ特徴で作ると、ほぼ同じ目的を二重に足すだけになります。そのため B_hat では、P_hat とは違う情報を見る必要があります。

候補見る情報役割
ブロック特徴セル内の2×2または4×4ブロック平均グラデーションや局所的なムラを拾う。
エッジ特徴セル内の明暗変化・境界方向輪郭が崩れすぎる組み合わせを避ける。
隣接連続性隣のセルとの境界差並べたときのつながりを評価する。
B_hat を P_hat と同じ YCbCr平均との差で定義すると、二観点最適化にならないので注意します。

7. 実装は交互最適化にする

π と λC を厳密に同時最適化するのは重すぎます。デモでは、交互最適化として近似します。

initialize π, λC

repeat:
  1. λC fixed: improve π by UI search
  2. π fixed: search λC in neighborhood
  3. evaluate J(π, λC)
  4. accept if J improves
  5. if stuck, shake π slightly

keep best (π, λC)

現状のデモに入っている「複数λを試して選ぶ」方法は、この交互最適化へ向かうためのプロトタイプとして扱います。

8. ブロックモデル再配置 z との関係

合成駒を作ったあとに、各画像に対して合成駒を置き直す変数 z または ρ を入れることもできます。ただしこれは UIサーチ本体とは別段階として整理します。

generation:
  min_{π, λC} J(π, λC)

re-placement:
  min_{ρ} R(π, ρ)

つまり、まず UIサーチ + YCbCr + 正則化付きλ探索で合成駒集合を作り、その後にブロックモデルで再配置補正を行う、という二段構成にします。

この分離により、「合成駒として良いか」と「絵として自然に戻せるか」を混同せずに実装できます。
